# Vector search — Milvus collection lifecycle and graceful degradation

## Milvus search graceful degradation

**ARTIFACT-INV-MILVUS-SEARCH-GRACEFUL.** `milvusClient.SearchVectorsInCollection` in `pkg/repository/vector.go` MUST return `nil, nil` (empty results, no error) when `HasCollection` returns false for the target collection, rather than propagating an error. The insert path (`InsertVectorsInCollection`) already auto-creates the Milvus collection on the first write via `CreateCollection`, so re-processing KB files through the standard ingestion pipeline naturally recovers the collection and its embeddings. Hard-erroring on search when a collection is missing causes cascading failures in downstream tool callers: a `search-chunks` consumer receives a gRPC error, retries with different parameters, each retry consumes seconds of upstream API quota, and the call eventually times out. With graceful degradation, `search-chunks` returns 0 chunks (the data simply isn't indexed yet) and the caller can fall back to other tools or an alternate context source. A warning log (`"Milvus collection does not exist, returning empty results"`) is emitted at `zap.Warn` for operational visibility without polluting error-level dashboards.

**Why collections can go missing.** Milvus collections live in a Docker named volume (`milvus-data`). They can disappear due to: (a) volume recreation during local development (`make down` removes volumes), (b) Milvus version upgrades that require re-indexing, (c) failed ingestion where Postgres chunk records were committed but the Milvus insert was not yet reached, (d) manual Milvus maintenance. The Postgres `chunk` table remains the source of truth for which chunks exist; Milvus is a derived index that can always be rebuilt from the chunk embeddings.

## File chunk integrity probe

**ARTIFACT-INV-FILE-CHUNK-INTEGRITY-PROBE.** `PrivateHandler.CheckFileChunkIntegrityAdmin` in `pkg/handler/private.go` is a read-only cross-datastore consistency probe: given a `file_uid`, it reports whether the file's `process_status` is actually backed by chunks in PostgreSQL, vectors in Milvus, and a `converted_file` row. It MUST:

1. **Return OK with a classification rather than a gRPC error.** A deleted/never-existing file row produces `IntegrityState = INTEGRITY_STATE_FILE_NOT_FOUND` in the success body, never `codes.NotFound`. Callers route entirely off `recommended_action`; mixing tombstone signals into the gRPC code would force every consumer to disambiguate "file genuinely gone" from "RPC plumbing failure."
2. **Probe in the cheapest order.** PG `chunk` count (one indexed `COUNT`), then Milvus per-`file_uid` vector count (gated by collection existence and the legacy-schema short-circuit in `milvusClient.CountVectorsForFileUID` — legacy collections without the `file_uid` metadata field are reported as `MISSING_MILVUS` so the caller triggers reprocess and lands the file under the new schema), then PG `converted_file` row presence. Stop on the first drift signal.
3. **Treat the chunk-rows-vs-vectors comparison as strict equality.** A non-zero PG chunk count with a non-equal Milvus vector count for the same `file_uid` is `MISSING_MILVUS` regardless of direction. A tolerance ("vectors >= chunks − 1") would mask half-completed re-ingestions where some chunks were dropped from Milvus during a rolling restart but PG still advertises the old count.
4. **Stay idempotent and safe to call concurrently** for the same `file_uid`. The probe is read-only; the recovery action is `RECOMMENDED_ACTION_REPROCESS_FILE`, which the caller invokes via the separate `ReprocessFileAdmin` RPC. `ReprocessFileAdmin` already short-circuits in-flight reprocesses, so concurrent probe-then-reprocess from N consumers is safe.

The probe does not assume a specific consumer; the operational context (who calls it, when, and what drift mechanisms motivate the call) lives with the consumer.
