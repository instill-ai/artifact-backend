# Storage streaming — GCS resume, MinIO range, transport tuning

## GCS in-activity resume

**ARTIFACT-INV-GCS-STREAM-RESUME.** `gcsStorage.StreamCopy` in `pkg/repository/object/gcs.go` MUST recover in-activity from mid-stream `io.ErrUnexpectedEOF` (and from "silent" truncations where the reader returns `io.EOF` but the copied byte count is strictly less than the declared object size) by re-opening the source reader through the caller-supplied `openAt(offset)` factory at the last known copied offset and resuming the write into the SAME GCS resumable-upload session (`storage.Writer` with `ChunkSize = 32 MiB` and `ChunkRetryDeadline` unmodified). The resume budget is `maxStreamResumes = 3` with a cancellable `streamResumeGap = 2 s` backoff between attempts; exceeding the budget closes the GCS writer (aborting the resumable session) and returns the wrapped `io.ErrUnexpectedEOF`. Non-EOF errors MUST NOT consume resume budget — they propagate unchanged so the caller (typically a Temporal activity) can decide on retry. Progress is reported via the caller-supplied `progressFn(copied int64)` with a monotonically non-decreasing cumulative counter across resumes; consuming activities rely on this signal to drive Temporal heartbeats, so a non-monotonic or segment-local counter would break the heartbeat contract under truncation. Tests: `pkg/repository/object/gcs_stream_test.go` (`TestStreamCopyWithResume_{CleanCopy,OneTruncation,ThreeTruncations,BudgetExhausted,NonEOFError,SilentTruncation,ProgressFnFires,ContextCanceledDuringBackoff}`).

## MinIO range offset-zero footgun

**ARTIFACT-INV-MINIO-RANGE-OFFSET-ZERO.** `minioStorage.GetFileReaderRange(ctx, bucket, filePath, offset)` in `pkg/repository/object/minio.go` MUST distinguish between `offset == 0` (whole-object read, no `SetRange` call) and `offset > 0` (partial read, `SetRange(offset, 0)` to express "bytes=N-" in HTTP Range syntax). The minio-go `GetObjectOptions.SetRange(0, 0)` translates to `Range: bytes=0-0` — a **single-byte** request, not a whole-object request — and silently truncating to 1 byte whenever the resume loop starts fresh would defeat the entire point of the StreamCopy helper. This is the "offset-0 footgun" documented in `pkg/repository/object/gcs_stream_test.go` (`TestMinIO_GetFileReaderRange_Offset0Footgun`). Any refactor that collapses the two branches into a single unconditional `SetRange(offset, 0)` call is a regression.

## MinIO client keep-alive (TODO)

**ARTIFACT-INV-MINIO-CLIENT-KEEPALIVE(TODO).** The MinIO client used by the artifact worker (constructed in `pkg/service/service.go` via `minio.New`) does not currently tune the underlying `http.Transport` keep-alive / idle-connection settings. Empirical observation on multi-GiB video uploads (`col-UvvNPpj04N`, IVOD long-form videos) shows that long-running `GetObject` bodies occasionally end with a clean but short-read — the working theory is an idle connection reaped mid-stream by the MinIO server's keep-alive timeout while io.Copy is still draining. Until this is investigated and tuned, the in-activity resume loop (`ARTIFACT-INV-GCS-STREAM-RESUME`) is the safety net. The follow-up work SHOULD: (a) set an explicit `http.Transport.IdleConnTimeout` and `ResponseHeaderTimeout` on the MinIO client, (b) measure the server-side keep-alive envelope MinIO's chart ships with, and (c) add a regression log-assertion that no more than one resume fires per multi-GiB upload in a soak run. Once resolved, this TODO invariant MUST be replaced with a concrete `ARTIFACT-INV-MINIO-CLIENT-KEEPALIVE` invariant that pins the chosen settings and references the production soak results.
